# Titan Setup — Developer Guide

## What This Is
Bash installer that transforms fresh Ubuntu into a complete AI dev workstation.
Installs 150+ CLI tools, Claude Code, services (Letta, n8n, Ollama), and deploys
`dot-claude/` → `~/.claude/` (settings, hooks, skills, rules, agents).

## Architecture
Use this map to find the right file. Go directly to it instead of grepping the project.

```
titan-setup.sh          ← assembled monolith (DO NOT edit — generated by just build)
lib/00-header.sh        ← shebang, version, self-materialization
lib/01-common.sh        ← helpers: ok(), warn(), fail(), section(), phase_done/mark(), apt_update()
lib/02-cli.sh           ← CLI option parsing (--name, --mode, --dry-run, --minimal, etc.)
lib/03-vps-reexec.sh    ← VPS sudo/user creation
lib/04-vps-harden.sh    ← Tailscale, SSH hardening, fail2ban, auditd
lib/05-prerequisites.sh ← system packages, journald limits, fonts
lib/06-package-managers.sh ← rustup, uv, bun, Go, mise, Docker
lib/06b-repo-files.sh   ← inline config stashing (TITAN_REPO_FILES)
lib/07-tools-python-js.sh ← Python tools (uv), JS tools (bun)
lib/08-tools-letta.sh   ← Letta server, Ollama, better-ccflare
lib/09-tools-rust-go.sh ← Rust tools (cargo), Go tools, binary downloads
lib/10-claude-code.sh   ← Claude Code CLI install
lib/11-deploy-config.sh ← Deploy ~/.claude/ from dot-claude/
lib/12-plugins.sh       ← Claude Code plugins
lib/13-plugins-letta-ctrl.sh ← Letta control plane web UI
lib/14-plugins-cleanup.sh ← Plugin cleanup/summary
lib/15-shell-integration.sh ← Shell config, PATH, verification
```

**Build:** `just build` assembles lib/*.sh → titan-setup.sh. Edit lib/ files, not titan-setup.sh.
**Test:** `just test` runs all bats tests. `just check` = full CI (lint+fmt+build+test+smoke).
**Smoke:** `just smoke` = `bash titan-setup.sh --dry-run --mode desktop --name test`

## set -e Safety — CRITICAL
This project uses `set -euo pipefail`. Any unguarded non-zero exit silently kills the entire script.
This has caused regressions 3 times. The `test/set-e-safety.bats` suite catches violations.

**Commands that MUST be guarded:** network calls (curl, wget), package installs (apt, cargo install),
service operations (systemctl enable/start), file downloads, and API calls.

**Commands that do NOT need guarding:** echo, mkdir -p, local variable assignment, printf, cat of known files.

```bash
# SAFE patterns:
curl -fsSL ... | bash || true                              # swallow failure
curl -fsSL ... && ok "done" || warn "failed"               # report outcome
VAR=$(curl -s ... | jq -r '.tag' || true)                  # guard pipeline
if curl -fsSL ... | bash; then ok "yes"; else warn "no"; fi # conditional

# DEADLY — these kill the script under set -e:
curl -fsSL ... | bash                     # network fail → exit 1
sudo apt-get install -y pkg               # missing pkg → exit 1
VAR=$(curl -s ... | jq -r '.tag')         # pipefail → exit 1
```

Multiline: guard goes on the FINAL line of a pipe/continuation chain, not the first.

**GitHub version fetching:** Use `_gh_latest_tag "owner/repo"` (defined in lib/09-tools-rust-go.sh).
It uses HTTP redirect following (zero API calls) instead of api.github.com (60 req/hr limit).

## When `just test` Fails
1. Read the full bats output — it shows the failing assertion and line number
2. If `set-e-safety.bats` fires: you added an unguarded command — add `|| true` or `&& ok ... || warn ...`
3. If `shellcheck.bats` fires: fix the warning; add disable comments only with justification
4. If `smoke.bats` fires: run `just smoke` standalone to see full output
5. If a grep-based test gives false positives: exclude guarded lines (`||`, `&&`, `if`, trailing `|` or `\`)

## Bug Fixes Require Regression Tests
Every bug fix MUST include a test in `test/*.bats` that catches the class of bug.
Commit fix + test together. Verify the test fails *before* your fix, passes *after*.
Reduce false positives in grep-based tests by excluding guarded lines.

## Adding a New Tool
1. Find the right lib/ file from the Architecture map (usually lib/07, lib/09, or lib/10)
2. Follow the existing pattern: check if installed → install → verify → ok/warn
3. Guard all install commands (see set -e Safety above)
4. Run `just build && just test`

## Helpers (lib/01-common.sh)
- `ok "msg"` / `warn "msg"` / `fail "msg"` — colored status output (checkmark/warning/X)
- `section "Phase N — Name"` — cyan phase banner
- `phase_done "phaseN"` / `phase_mark "phaseN"` — idempotent checkpoint system (~/.titan-progress/)
- `apt_update` — cached apt-get update (runs once per session via `_APT_UPDATED`)
- `check_port 8080 "service"` — warn if port already bound
- `run_q cmd...` — run quietly (stdout/stderr suppressed)
- `_gh_latest_tag "owner/repo"` — get latest release tag via HTTP redirect (no API call)

## Conventions
- **Variables set by CLI flags or environment:** `UPPERCASE` (DRY_RUN, MINIMAL, ENGINEER_NAME)
- **Variables internal to script logic:** `_PREFIXED` (_APT_UPDATED, _CARGO_LIST, _VER_DIR)
- **Local variables inside functions:** use `local` keyword, lowercase
- **Phases are idempotent:** each checks `~/.titan-progress/` before re-running; `--force-updates` overrides
- **Hooks (dot-claude/hooks/) must NOT use `set -euo pipefail`** — a crash kills the user's session
- **After editing any lib/*.sh file:** run `just build && just test` before committing

## dot-claude/ (deployed to ~/.claude/)
- `CLAUDE.md.tmpl` — global operating manual (applies to ALL projects, not just titan)
- `settings.json` — 73 deny rules, hooks, env vars, permissions
- `skills/` — 15 self-contained skill files
- `hooks/` — session-start, session-end, pre-compact, prompt-memory-inject
- `rules/` — 7 path-gated rules (shell, python, docker, terraform, security, memory, skill-authoring)
- `agents/` — researcher (Haiku), planner (Opus), reviewer (Sonnet)
- `commands/` — /ship, /scan, /review, /remember, /recall, /tools, /workspace-init, /handoff, /gh-action, /pack, /standup, /catchup
